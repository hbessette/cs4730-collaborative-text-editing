### Build configuration

All evaluation runs use the release build with debug logging enabled so that `LATENCY_SEND` / `LATENCY_APPLY` entries are written to log files:

```bash
cmake -B build -DCMAKE_BUILD_TYPE=Release -DENABLE_DEBUG_LOG=ON
cmake --build build -j$(nproc)
```

### Clock synchronisation

All nodes are NTP-synchronised. Microsecond-precision wall-clock timestamps (`CLOCK_REALTIME`) are used. Any sample with a negative delta (receiver timestamp earlier than sender timestamp, indicating residual NTP skew) is excluded from latency statistics.

### Repeatability

All experiments can be reproduced by running:

```bash
make eval-latency
make eval-convergence
make eval-concurrent
make eval-scalability
make analyze
```

See `scripts/cluster.conf` for the node list and `scripts/run_eval.sh` for full orchestration details.

## 3. Experiment 1 — Latency

### Methodology

Peer 0 (sender) inserts 1000 characters sequentially using a headless script generated by `scripts/automated_typing.sh`. All other peers are passive receivers. The experiment runs twice: once with 2 total peers, once with 5.

Each operation is logged at send time (`LATENCY_SEND ts_us=<epoch_us>`) and at apply time on each receiver (`LATENCY_APPLY ts_us=<epoch_us>`). The per-op latency is `apply_ts − send_ts` in milliseconds.

**Parameters:**

| Parameter | Value |
|-----------|-------|
| Operations | 1000 INSERTs |
| Send rate | ~100 ops/sec (10 ops per 100 ms sleep) |
| Sub-experiments | 2-peer, 5-peer |
| Repetitions | 1 each |

**Analysis:** `python3 scripts/analyze_results.py logs/`

### Results

| Config | Peers | Ops sent | Delivered | P50 (ms) | P95 (ms) | P99 (ms) | Drop % |
|--------|-------|----------|-----------|----------|----------|----------|--------|
| 2peer  | 2     |          |           |          |          |          |        |
| 5peer  | 5     |          |           |          |          |          |        |

*Populate this table by running `make eval-latency && make analyze`.*

### Expected behaviour

- P50 latency below 5 ms on a LAN (dominated by one UDP hop + CRDT apply time).
- P99 latency may spike if UDP queues back up under sustained 100 ops/sec load.
- Drop rate should be < 1%. Rates above 1% indicate the receiving pipeline cannot keep up and are flagged as a warning by `analyze_results.py`.
- 5-peer drop rate may be slightly higher than 2-peer because each sender hop fans out to 4 receivers instead of 1.

## 4. Experiment 2 — Convergence correctness

### Methodology

50 independent trials, each with a randomly selected peer count (3–5) and a unique seeded random INSERT/DELETE script per peer. All peers start writing with a small stagger (5 s startup delay) to exercise causal ordering. After all scripts finish, each peer sleeps 5 s for quiescence and then dumps its document.

A trial is **PASS** if all `peer_*_dump.txt` files are byte-for-byte identical; **FAIL** otherwise.

Scripts are generated by `scripts/gen_convergence_script.py` with `--insert-prob 0.7` (70% inserts, 30% deletes).

**Parameters:**

| Parameter | Value |
|-----------|-------|
| Trials | 50 |
| Peers per trial | 3–5 (uniform random) |
| Ops per peer | 200 |
| Insert probability | 0.70 |
| Base seed | 42 |
| Quiescence sleep | 5 s |

**Analysis:** `python3 scripts/analyze_results.py logs/`

### Results

| Metric | Value |
|--------|-------|
| Total trials | 50 |
| Passed | |
| Failed | |
| Pass rate | |

*Divergent trials (if any) are detailed below.*

| Trial | Peers | Ops | Seed | Failure reason |
|-------|-------|-----|------|---------------|
| — | — | — | — | — |

*Populate by running `make eval-convergence && make analyze`.*

### Expected behaviour

The RGA algorithm guarantees convergence for any sequence of concurrent inserts and deletes, provided all operations are delivered. On a LAN with no packet loss the expected pass rate is 100/100. Any failure indicates either a bug in the CRDT or undelivered operations (check the drop rate in Experiment 1). A `divergence_report.txt` is written to each failed trial directory for post-mortem inspection.

## 5. Experiment 3 — Concurrent writes

### Methodology

Identical to Experiment 2 except all peers start writing at `t=0` (no startup stagger), maximising simultaneous conflict. This stresses the `insertNode` tiebreaker logic by generating the highest possible rate of concurrent inserts targeting the same positions.

**Parameters:**

| Parameter | Value |
|-----------|-------|
| Configuration | 5 peers |
| Ops per peer | 200 |
| Start stagger | 0 ms |
| Quiescence sleep | 5 s |

### Results

| Config | Peers | Ops sent | Delivered | P50 (ms) | P95 (ms) | P99 (ms) | Drop % | Converged |
|--------|-------|----------|-----------|----------|----------|----------|--------|-----------|
| concurrent_5peer | 5 | | | | | | | |

*Populate by running `make eval-concurrent && make analyze`.*

### Expected behaviour

- Convergence: PASS. The RGA tiebreaker `(clock, siteID)` is a total order; every peer applies the same ordering for concurrent inserts anchored at the same `leftID`.
- Latency may be higher than Experiment 1 because every peer is simultaneously sending and receiving at peak throughput.
- Drop rate may be elevated under maximum contention. If drop rate exceeds 1%, increase the `--sleep-ms` parameter in `automated_typing.sh` to pace the send rate.

## 6. Experiment 4 — Scalability

### Methodology

Measures system performance as the number of peers grows from 2 to 10 in steps of 2. Each peer writes 100 ops at 5 ops/sec. Because the cluster has 5 nodes, configurations with more than 5 peers co-locate additional peers on existing nodes (round-robin, max 2 per node).

Metrics collected per configuration:

| Metric | How measured |
|--------|-------------|
| **Convergence time (ms)** | Wall time from first `LATENCY_SEND` to last `LATENCY_APPLY` across all peer logs |
| **Per-peer throughput (ops/sec)** | Total ops converged / convergence time / peer count |
| **CPU usage (%)** | Average `%CPU` from `/usr/bin/time -v` per peer |

A **bottleneck** is flagged if convergence time grows ≥ 2× faster than the peer count, or if any configuration exceeds 80% average CPU.

**Parameters:**

| Parameter | Value |
|-----------|-------|
| Peer counts | 2, 4, 6, 8, 10 |
| Ops per peer | 100 |
| Send rate | 5 ops/sec |
| Max peers per node | 2 |

### Results table

|-------|----------------|-----------------------------|-----------|
| 2     |                |                             |           |
| 4     |                |                             |           |
| 6     |                |                             |           |
| 8     |                |                             |           |
| 10    |                |                             |           |

*Populate by running `make eval-scalability && make analyze-scalability`.*

### Scalability chart

The chart `logs/scalability/scalability_chart.png` (generated by `python3 scripts/analyze_scalability.py logs/scalability/`) plots three series against peer count:

```
Y-axis (left):  Convergence time (ms)  — blue line
```

Generate it with:
```bash

    --csv logs/scalability/results.csv \
    --chart logs/scalability/scalability_chart.png
```

### Expected behaviour

- **Convergence time** should grow roughly linearly with peer count (each additional peer adds one more UDP fan-out recipient).
- **Per-peer throughput** should remain approximately constant (each peer's share of the work does not change as the cluster grows).
- **CPU** should remain well below 80% for all configurations; the CRDT apply loop is O(n) in document length, not peer count.
- A super-linear increase in convergence time would indicate that the `pendingOps_` retry loop is becoming a bottleneck (increasing causal-dependency depth under high concurrency).

## 7. Analysis and conclusions

### Summary

| Experiment | Expected result | Actual result |
|------------|----------------|--------------|
| Latency (2-peer P50) | < 5 ms | |
| Latency (5-peer P50) | < 10 ms | |
| Convergence pass rate | 50/50 (100%) | |
| Concurrent convergence | PASS | |
| Scalability bottleneck | None up to 10 peers | |

### CRDT correctness

The RGA algorithm provides **strong eventual consistency (SEC)**: any two peers that have received the same set of operations will have identical document states, regardless of delivery order. This is guaranteed by:

1. Unique `CharID` per character (Lamport clock + siteID — no coordination needed).
2. A deterministic total order on `CharID` used by `insertNode` to resolve concurrent inserts.
3. Idempotent apply: duplicate inserts and deletes are silently ignored.
4. A `pendingOps_` buffer that handles out-of-order UDP delivery without losing operations.

Experiment 2 (50-trial convergence) validates this guarantee empirically across randomised workloads with varying peer counts.

### Latency sources

End-to-end operation latency has four components:

| Component | Typical cost |
|-----------|-------------|
| UDP transmission (LAN) | < 1 ms |
| `receiveLoop` dequeue | < 0.1 ms |
| `crdtLoop` dequeue + mutex acquire | < 0.1 ms |
| `insertNode` (O(document length) in worst case) | < 1 ms for documents < 10 000 chars |

The dominant factor at scale is the O(n) `visibleAt` and `insertNode` scan. For the document sizes used in evaluation (≤ 1 000 characters) this is negligible. For very large documents, the index (`index_`) reduces lookup of a known `CharID` to O(1), but the visible-position scan remains O(n).

### Limitations and future work

- **UDP delivery is unreliable.** The current design has no retransmission. Operations dropped by the network are lost permanently, causing divergence. A future version could add operation-level ACKs or use a TCP transport for the operation stream.
- **O(n) visible-position scan.** `visibleAt` and `visibleIter` iterate the full sequence to find a visible position. A skip list or augmented tree would reduce this to O(log n).
- **No persistence.** Document state is lost when all peers disconnect. A write-ahead log would enable recovery.
- **No access control.** Any peer that knows the port can join and edit. Authentication is out of scope for this prototype.
